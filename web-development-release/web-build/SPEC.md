## Overview

The ArchUI web app's source code is not centralised in a single `src/` directory. Instead, each GUI spec module generates its own component files in its `resources/` when `update-resources` runs. `web-build` is responsible for knowing where all these files live, assembling them into a buildable project, and producing the final `dist/` output.

This is a fundamentally different build model from traditional frontend projects — the build configuration adapts as modules are added or removed from the architecture.

## Module-Source Manifest

`resources/web-manifest.yaml` is the machine-readable map of every module that contributes source files to the web build:

```yaml
# web-manifest.yaml
# module-path: list of files it generates relative to its own resources/
gui/components/module-node/default:
  - src/components/ModuleNodeDefault.tsx
  - src/components/ModuleNodeDefault.module.css

gui/components/module-node/command-bar:
  - src/components/CommandBar.tsx
  - src/components/CommandBar.module.css

gui/screens/canvas:
  - src/components/canvas/CanvasPage.tsx
  - src/components/canvas/CanvasPage.module.css
  - src/store/canvas.ts

web-development-release/web-dev:
  - src/main.tsx
  - src/App.tsx
  - src/types.ts
  - src/design-tokens.css
  - vite.config.ts
  - tsconfig.json
  - package.json
```

### Manifest maintenance rules

- When a module's `update-resources` generates new files, it must also update this manifest.
- When a module is added or removed from the architecture, the manifest is updated accordingly.
- Each entry maps the module path to the files it owns — no two modules may claim the same output file.
- The manifest is the source of truth for the build pipeline; the build script never scans directories.

## Build Pipeline

The build uses a **stage-then-compile** approach:

```
1. Read web-manifest.yaml
2. For each (module, files) entry:
   a. Resolve source path: <module>/resources/<file>
   b. Copy or symlink to staging area: build/stage/<file>
3. Run Vite with root = build/stage/
4. Output to dist/
```

This keeps each module's source isolated in its own `resources/` while presenting a flat `src/` layout to Vite.

### Dev build (HMR mode)

In development, the staging area uses symlinks instead of copies, so Vite HMR detects changes in any module's `resources/` immediately without a re-stage step:

```bash
npm run dev          # stage with symlinks, then vite dev
```

### Production build

In production, the staging area uses copies (no symlinks), and Vite runs with full optimisation:

```bash
npm run build        # stage with copies, then vite build
```

## Build Configuration Ownership

Build configuration files are generated by `web-dev/resources/`:

| File | Description |
|---|---|
| `vite.config.ts` | Vite config — uses `web-manifest.yaml` to set up aliases |
| `tsconfig.json` | TypeScript strict mode config |
| `package.json` | Dependency manifest |
| `src/main.tsx` | React entry point |
| `src/App.tsx` | Root component |

These files are listed in the manifest under `web-development-release/web-dev`. They define the project skeleton; all other modules add components and utilities on top.

## Adding a New Web Module

When a new GUI module is created that generates web source files:

1. Add an `update-resources` command to the module.
2. After `update-resources` runs, update `web-manifest.yaml` with the new module and its output files.
3. Run `npm run build` to verify the new files integrate correctly.
4. Adjust `vite.config.ts` or `tsconfig.json` if new path aliases are needed.

## Adjusting the Build After Architecture Changes

If modules are restructured (split, merged, renamed):
1. Update `web-manifest.yaml` to reflect the new module paths and file ownership.
2. Re-run `npm run build` and resolve any import errors.
3. The test suite (`web-test`) must still pass — if not, adjust the affected module's generated files.
