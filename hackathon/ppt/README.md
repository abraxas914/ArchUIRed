## 概述

PPT 模块负责黑客松演示文稿的全部规范，包括叙事结构、每页内容要点、视觉风格与排版约束。

## 内容要点

### 问题

代码和文档是割裂的，导致三个核心痛点：

- **文档与代码脱节** — 文档写完就过时，代码改了文档没跟上，新人看文档写出的代码和实际架构不一致
- **上下文冗余** — AI agent 生成代码时需要加载大量文档，长文档中大部分内容与当前任务无关，浪费 token 和注意力
- **知识不可复用** — 同样的登录逻辑、同样的鉴权方案，每个项目重新写一遍文档、重新让 AI 理解一遍

### 解决方案

**代码与文档绑定，文档模块化，模块直接生成代码：**

- 把代码变成文档 — 每个功能模块（登录、支付、鉴权…）对应一个文档模块
- 文档拆成模块树 — 层级化、可嵌套，每个模块独立可理解
- 模块可复用可替换 — 需要登录功能？直接引用登录模块文档，AI agent 根据文档生成对应代码
- 人类用画布编排模块关系，AI agent 按需加载模块上下文生成代码
- 文件系统即知识库，git 天然版本管理，文档改了代码跟着更新

### 竞品对比

#### Monorepo 工具（Nx、Turborepo）

Monorepo 工具只编排**代码**的构建和依赖，文档和代码仍然是割裂的：

| 维度 | Monorepo（Nx/Turborepo） | ArchUI |
|---|---|---|
| 编排对象 | 只编排代码（构建、缓存、任务） | 文档与代码一体编排 |
| 模块化 | 代码包模块化，文档另外维护 | 文档模块本身定义代码结构和依赖 |
| 依赖管理 | package.json 管代码依赖 | UUID 链接管模块关系，包含代码编排 |
| AI 上下文 | AI 能查代码依赖图，但不懂业务意图 | AI 从文档模块获取意图+结构，直接生成代码 |
| 知识复用 | 代码包可复用，但需求文档每次重写 | 文档模块可复用，需求+代码一起迁移 |
| 需求变更 | 改代码，文档不跟着更新 | 改文档模块，AI 同步受影响的代码 |

**核心差异：Monorepo 只管"怎么建"，ArchUI 从文档层就定义了"建什么" — 文档即编排。**

#### LLM Wiki（Karpathy 提出）

LLM 全权维护一组 Markdown 页面，人类提供原始资料并提问，LLM 负责摘要、交叉引用、归档。依赖 index.md 做全局导航，log.md 记录操作日志。

#### RAG

通过 embedding 向量检索文档片段，拼接到 prompt 中供 LLM 生成回答。

#### 综合对比

| 维度 | RAG | LLM Wiki | Monorepo | ArchUI |
|---|---|---|---|---|
| 管什么 | 文档检索 | 文档生成与维护 | 代码构建编排 | 文档+代码一体编排 |
| 知识组织 | 向量切片，碎片化 | 扁平页面 + index.md | 代码包 + 依赖图 | 模块树，层级化，文档绑定代码 |
| 上下文控制 | 检索质量不可预测 | 规模大后索引崩溃 | 只管代码，不管文档上下文 | 按需加载模块，渐进披露 |
| 代码生成 | 片段拼接 prompt | 不生成代码 | 不生成代码，只编排构建 | 模块即 spec，AI 直接生成代码 |
| 可复用性 | 每次重新检索 | 不可即插即用 | 代码包可复用，文档不可 | 文档模块可复用，代码跟着走 |
| 可维护性 | 重建索引成本高 | 幻觉固化为"事实" | 文档与代码脱节 | 文档改了代码跟着更新 |

**AI Developer Infrastructure（AI 开发基础设施）**

> AI agent 能读懂代码了（Monorepo），但还读不懂文档 — 这是一个尚未被解决的基础设施层。

ArchUI 横跨四个市场：

| 市场 | 代表玩家 | ArchUI 切入点 |
|---|---|---|
| AI 辅助开发 | Cursor, GitHub Copilot, Devin, Windsurf | 从文档侧解决 AI 上下文问题，跟 AI coding 工具互补 |

| 文档即代码 | Backstage, Docusaurus, Mintlify | 文档与代码绑定，但 ArchUI 更进一步 — 文档生成代码 |

### 路线图

**Phase 1：代码开发编排（当前）**

ArchUI 从代码开发领域切入 — 文档模块化、AI agent 按需加载、模块即 spec 直接生成代码。

**Phase 2：模块广场（近期）**

建立模块共享生态 — 用户可以将自己的模块发布到广场，也可以从广场获取他人的模块：

| 能力 | 说明 |
|---|---|
| **发布模块** | 一键将本地模块树发布到广场，保留结构和链接关系 |
| **发现与获取** | 按领域/标签浏览模块，一键拉取到本地项目中 |
| **版本管理** | 模块发布后可迭代更新，引用方可选择是否升级 |
| **组合复用** | 拉取一个"登录模块"文档，AI agent 直接基于它生成代码 — 不用从零写 spec |

**为什么模块广场是关键一步：**
- 解决冷启动 — 新用户不需要从空白开始，拉取现成模块即可开工
- 网络效应 — 模块越多，平台越有价值；用户既是消费者也是生产者
- 商业化基础 — 高质量模块可收费，广场抽成（类似 npm + Vercel 模板市场）

**类比：**
- ComfyUI 的工作流共享社区 — 但 ArchUI 共享的是结构化知识模块，不只是节点配置
- npm/pip — 但共享的是文档+spec，不只是代码包
- Figma Community — 但面向的是 AI agent 消费，不只是人类浏览

**Phase 3：扩展到创作编排（未来）**

模块化编排的逻辑不限于代码。任何有"多步骤 + 多工具 + 上下文传递"特征的创作流程，都可以用 ArchUI 的模块树 + 画布来编排：

| 领域 | 类比 | 模块化价值 |
|---|---|---|
| AI 漫剧制作 | 策划→角色→场景→动画→后期，5 个阶段 6+ 工具 | 每个阶段是一个 SPEC 模块，工具链和提示词在 resources 里，流程可复用可迭代 |
| AI 视频生产 | 脚本→分镜→素材→剪辑→配音 | 同样的模块树逻辑，换工具节点即可 |
| 课程/教程制作 | 大纲→章节→素材→录制→发布 | 知识结构天然是模块树 |

**为什么能扩展：**
- ArchUI 编排的是**知识和流程**，不是代码本身 — 代码只是模块的一种输出物
- 创作流程的痛点跟代码开发一样：上下文割裂、工具间交接靠人工、流程不可复用
- 节点画布（ComfyUI 风格）天然适合可视化创作流水线

**市场空间：**
- AI 内容创作市场 2025 年 $196 亿，2030 年预计 $801 亿（CAGR 32.5%）
- AI 视频生成市场 2025 年 $7.9 亿，2033 年预计 $34 亿（CAGR 20.3%）
- 2026 年预计 50% 的创意素材将由 AI 生产或辅助生产
- ComfyUI 已验证节点化创作工具的用户需求，但仅限图片/视频生成环节，缺乏全流程编排

**一句话讲未来：**

> 今天我们用模块编排代码，明天用同样的模块编排一切创作流程 — 从 AI 开发基础设施到 AI 创作基础设施。
